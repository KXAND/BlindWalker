# ADR 0016: Door Interaction

**日期**: 2026-07-15
**状态**: Accepted

---

## 背景

玩家需要能与门互动，打开后进入对应空间，并能再次互动关门。现有通用 `Interactable` 已经覆盖“进入范围后按 E 触发”的交互模型，但门有额外约束：提示应来自门把手，开关门需要处理碰撞、开向、音效时机和玩家占位。

## 决策

新增可复用的门交互行为，但本轮只配置明确目标门：`scenes/Buildings/Department.tscn` 中的标准门，以及 `scenes/Home/PlayerHome.tscn` 中的主 `Door`。其他使用同一门资源的实例本轮不自动获得互动，避免顺手改变楼梯间等未确认门的行为。

门交互继承通用可互动对象规则：玩家可以在未显示提示时盲按 E 触发当前焦点；提示 UI 只有门把手或门把手附近锚点被手触/杖触显影后才显示。显影目标不使用整扇门板。

门通过实例可配置的开门角度切换开关状态，默认开角为 90 度，实例可覆盖为 -90 度以适配不同朝向。推荐围绕门的铰链/pivot 旋转门体，不使用真实物理门。

关门前必须检查玩家占位。如果玩家处于门扇闭合区域或路径内，关门被拒绝，门保持打开，避免把玩家卡进门体。盲杖不作为拒绝关门的硬条件，但实现阶段必须验证不会导致输入或盲杖状态卡死。

本轮只做开门/关门音效，不触发剧情。开门音效在开门开始的瞬间播放；关门音效在门完全关上的瞬间播放。

## 考虑的替代方案

| 方案 | 否决理由 |
|------|---------|
| 直接给共享门 prefab 的所有实例启用互动 | 会让楼梯间等未确认门获得新行为，扩大改动范围 |
| 用整扇门作为提示显影目标 | 玩家触到任意门板就能看到互动提示，不符合“把手是可操作部件”的语义 |
| 用真实物理门 | 调试成本高，容易引入挤压、卡住、抖动等问题；当前只需要确定性的开关状态 |
| 关门时不检查玩家占位 | 容易把玩家或盲杖卡进门体，破坏移动和感知系统 |
| 门互动触发剧情 | 开关门是世界状态反馈，不应默认锁输入或进入叙事流程 |

## 验收要求

实现阶段不能只启动游戏确认无报错。需要在 `scripts/debug/` 添加临时测试脚本，控制玩家进入目标门互动范围、触发互动并跟踪 Godot 输出，至少确认：

- 目标门按 E 后 pivot 角度从关闭状态切换到打开状态。
- 门打开后玩家可以穿过门洞。
- 再次互动会尝试关门，玩家占位时关门被拒绝。
- 玩家不占位时门能关上。
- 门提示显影目标是门把手或门把手附近锚点，不是整扇门板。
- 开门音效在开门开始时触发，关门音效在门完全关上时触发。
- 本轮未配置的其他门不会获得互动行为。
- 临时测试脚本完成后必须移除。
