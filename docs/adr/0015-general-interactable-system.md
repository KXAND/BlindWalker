# ADR 0015: General Interactable System

**日期**: 2026-07-14
**状态**: Accepted

---

## 背景

现有交互只覆盖 NPC 距离提示，无法统一表达开门、查看物品、收集任务物品、扶住护栏等行为。BlindWalker 的交互还必须遵守感知体验：玩家可以在未看到 UI 时尝试互动，但 UI 不能替玩家发现未被盲杖或手触感知到的物体。

## 决策

### 统一可互动对象

引入通用 `Interactable` 模型。门、家中物品、商品、护栏、NPC 都应迁移为可互动对象，不再保留 NPC 专用交互信号或兼容层。

每个可互动对象自己决定当前是否可互动、显示什么提示、互动后产生什么效果。中心交互系统只负责候选管理、焦点选择、提示显示和调用互动，不写门、商品、护栏、NPC 的业务分支。

可互动对象的粒度按“玩家可触发的动作单位”划分，而不是按视觉物体整体划分。一个货架上的不同商品、左右护栏段、门板和门把手如果动作不同，就应拆成不同的可互动对象。

### 候选、焦点与提示

可互动对象使用触发范围决定互动资格。玩家进入范围后，该对象成为候选；玩家离开范围后，该对象不再是候选。

当前只允许一个互动焦点。焦点选择不依赖显影，主要由面向程度、距离和互动优先级决定。互动优先级只用于近似并列时打破冲突，不应覆盖玩家明显面向的对象。

互动提示文本由具体对象提供，例如“按 E 打开”“按 E 查看”“按 E 扶住”。交互系统不拼接、不猜测提示语义。

互动提示使用屏幕 UI 表现为悬浮在对象上方。对象应提供提示锚点；交互系统将锚点投影到屏幕坐标。缺少锚点或显影目标时允许开发期 fallback，但必须输出 warning，正式内容应显式配置。

### 显影只控制 UI

互动资格和互动触发不依赖显影。玩家可以在未显示提示时按 E 触发当前互动焦点。

互动提示可见性依赖显影。只有当前互动焦点对应的显影目标已被感知时，才显示它的提示。显影目标应是互动动作对应的具体部件或最近语义父物体，而不是随意扩大到整栋物体。例如门把手互动只应在门把手被显影后显示提示，门板其他位置被显影不应让门把手提示出现。

显影判断由感知系统提供世界空间语义查询接口，而不是通过屏幕像素、材质状态或后处理结果反推。本轮由 `TouchMemorySystem` 暴露通用感知可见性查询，未来如出现多种感知来源，可再独立为感知可见性服务。

唯一焦点原则必须成立：E 触发的对象和 UI 提示对象必须是同一个对象。不能因为当前焦点未显影，就改为显示另一个已显影候选的提示。

### 可达性与二次校验

默认要求互动焦点视线无遮挡。系统从玩家头部或相机位置向提示锚点做物理射线；如果第一个命中对象属于该可互动对象、其显影目标或它们的相关节点，则认为无遮挡；如果先命中墙、货架背板、其他物体等，则认为被遮挡。

视线无遮挡不是感知可见性，不会单独让 UI 显示。它只用于防止隔墙、隔货架误互动。护栏等贴身对象可以配置关闭视线遮挡要求。

按 E 触发时必须做二次校验：对象仍有效、玩家仍在触发范围内、对象仍允许互动、仍满足基础面向条件，并在需要时满足无遮挡条件。显影不参与触发校验。

### 输入规则

互动键为 E，是普通 gameplay 输入，和 W、SHIFT、SPACE 等同级。

当 gameplay 输入整体不可用时，E 也不可用，例如死亡、Loading、叙事锁输入、摔倒、滚落和起身期间。教程不锁输入，因此不阻止互动。QTE 只接管 SHIFT + SPACE 的挽回含义，不提升或压低 E 的优先级。

### 互动效果

一次性对象效果由对象自己处理，例如开门、切换自身状态、播放自身音效。

会改变玩家长期 gameplay 状态或跨系统状态的互动，应通过事件或请求交给负责系统处理，避免对象直接修改玩家内部状态。

物品查看或自言自语类互动应触发普通剧情演出，和其他剧情一样默认锁定输入、显示对话框并支持跳过。

### 任务物品

不引入购物篮 UI、数量系统、错误商品系统、丢弃或放回功能。场景里可以有许多商品可互动，但只有配置为任务物品的对象会被收集。

任务物品用 `item_id` 标识。`GameState` 使用 set 语义记录已收集的任务物品，不记录数量。是否播放“找到了”等剧情反馈由具体任务物品对象配置，不写死在 `GameState` 中。

### 扶栏辅助

护栏互动采用按 E 切换状态：按一次进入扶栏辅助，再按一次、离开护栏范围、护栏失效、摔倒或进入其他不允许状态时退出。

扶栏辅助状态由玩家或步态系统持有，只在当前护栏有效范围内生效。扶栏辅助下楼梯时等效满足谨慎条件，因此不会因为未按 SHIFT 而摔倒。它不改变移动速度，也不免疫撞墙、车辆、深坑等其他危险。

进入和退出扶栏使用音效反馈，不触发剧情锁输入。正式音效缺失时，可以将现有音效临时映射为占位。扶栏对象至少应配置进入和退出音效 ID；其他互动对象也可以配置可选互动音效。

## 非目标

本 ADR 不实现完整购物篮、库存、数量、放回、丢弃或错误商品系统。

本 ADR 不新增非锁输入的轻量物品描述系统；物品描述统一走普通剧情演出。

本 ADR 不要求所有旧场景一次性完成内容配置，但新交互模型中 NPC 专用交互信号不保留兼容层。

## 验收要求

实现阶段不能只启动游戏确认无报错。需要在 `scripts/debug/` 添加临时测试脚本，控制玩家进入触发范围、转向、触发显影和按 E，并跟踪 Godot 输出，至少确认：

- 未显影时，当前焦点可被 E 触发，但不显示提示。
- 显影目标被显影后，当前焦点显示对象配置的提示。
- 显影门板其他位置不会让门把手互动提示显示。
- 多候选时 E 触发对象和 UI 提示对象始终一致。
- 隔墙或被其他物体遮挡时，默认不可成为互动焦点。
- NPC 互动走通用 Interactable，不再依赖旧 NPC 交互信号。
- 任务物品收集写入 `GameState` 的 set 记录，不重复计数。
- 扶栏进入/退出播放音效，并在扶栏有效范围内让下楼梯等效满足谨慎条件。
- 临时测试脚本完成后必须移除。

## 考虑的替代方案

| 方案 | 否决理由 |
|------|---------|
| 纯相机中心射线选中 | 对门把手、商品、护栏等对象过于苛刻，玩家需要像 FPS 一样精确瞄准 |
| 纯范围互动并总是显示提示 | 会让 UI 替玩家发现未感知物体，破坏黑暗探索体验 |
| 为 NPC 保留专用交互系统 | 会让 NPC、门、商品、护栏形成两套交互模型，增加长期复杂度 |
| 由中心系统按类型分发互动效果 | 门、商品、护栏、NPC 状态差异大，中心系统会快速膨胀 |
| 互动提示选择已显影候选而非当前焦点 | 会出现“看到 B 的提示却触发 A”的严重一致性问题 |
