# ADR 0013: QTE and Narrative UI Priority

**日期**: 2026-07-14
**状态**: Accepted
**修正**: ADR-0009, ADR-0011

---

## 背景

ADR-0011 引入了失衡踉跄和挽回 QTE，但当前 QTE UI 更接近调试面板：显示失衡百分比、剩余时间和恢复状态。正式体验中，玩家只需要明确知道此刻必须按住哪个组合键，而不应阅读数值。

ADR-0009 已经规定叙事序列支持 `SPACE` 跳过当前行、`ESC` 跳过整个序列，但当前对话框没有把这些能力以 UI 提示表达出来。跳过提示不能写死在台词文本中，因为跳过能力属于叙事系统和输入系统，而不是剧情文案的一部分。

此外，QTE、叙事播放、摔倒和死亡之间需要明确优先级，避免同时显示多个互相冲突的 UI。

## 决策

### QTE 提示正式化

失衡踉跄中的 QTE UI 从调试状态面板改为正式的紧急动作提示。

QTE 提示只显示按键组合：

```text
SHIFT + SPACE
```

QTE UI 不显示：

- 失衡百分比。
- 剩余时间。
- “恢复中”或“失衡中”等状态文本。

未按住 QTE 时，提示使用黄色，并通过轻微大小缩放或闪烁表达紧急感。

按住 QTE 时，提示改为白色或绿色等稳定高亮状态，用于明确区分“已经按住”和“尚未按住”。该状态只表达玩家正在执行挽回动作，不暴露数值进度。

QTE 提示位置采用屏幕中心偏下，而不是底部字幕区。底部区域优先留给叙事字幕。

### 对话跳过提示

叙事对话框应显示系统级跳过提示，而不是把提示写进 `NarrativeLine.text`。

默认提示语义为：

- `SPACE`：跳过当前叙事行。
- `ESC`：跳过整个叙事序列。

该提示属于 UI 层和输入层的表达，不能污染剧情文本。当前阶段不为每个叙事序列或叙事行新增跳过提示字段；跳过能力仍由叙事系统统一提供。

未来如果支持改键，提示文本应从统一输入映射或统一配置派生，而不是由剧情资源逐条配置。

### UI 与状态优先级

如果玩家已经处于叙事播放中，不应触发新的失衡 QTE。叙事播放期间产生新 QTE 通常意味着 gameplay 入口没有被正确阻止。

如果玩家在开始叙事前已经处于 QTE 状态，QTE 不会被叙事取消、隐藏或暂停。叙事锁定输入后，玩家无法继续执行 QTE，失衡状态会自然推进，并可能进入摔倒。

死亡/失败流程优先于叙事流程。若 QTE 失败导致摔倒，并进一步导致 HP 归零，死亡/失败 UI 必须中断叙事 UI。

摔倒流程优先于叙事流程。若 QTE 失败导致摔倒但玩家未死亡，叙事也应中断，由摔倒镜头和身体状态反馈接管。

本 ADR 将 `lock_gameplay` 的语义明确为：阻止叙事期间产生新的 gameplay 后果，但不冻结叙事开始前已经存在的危险状态。

## 非目标

本 ADR 不实现教学 UI。

例如“玩家第一次摔倒后提示什么情况下会摔倒”属于上下文教学或失败教学系统，应由后续独立 ADR 讨论。该系统需要单独决定触发条件、是否只出现一次、是否暂停游戏、如何关闭，以及如何与叙事、失败和 QTE UI 协同。

本 ADR 不修改 QTE 的判定数值，也不修改摔倒、滚落或伤害规则。

本 ADR 不引入每段剧情可自定义的跳过提示字段。

## 验收要求

实现阶段不能只启动游戏确认无报错。需要在 `scripts/debug/` 添加临时测试脚本，控制或模拟对应状态并跟踪 Godot 输出，至少确认：

- 失衡 QTE 时 UI 只显示 `SHIFT + SPACE`。
- 未按住和按住 QTE 时 UI 有明确视觉差异。
- QTE UI 位于屏幕中心偏下，不占用字幕底部区域。
- 叙事播放时对话框显示 `SPACE` 跳当前行和 `ESC` 跳整个序列的提示。
- 跳过提示不写入剧情文本。
- 叙事播放期间不会新触发 QTE。
- 叙事开始前已有 QTE 时，QTE 会自然继续并可能失败。
- 摔倒或死亡会中断叙事 UI。
- 临时测试脚本完成后必须移除。

## 考虑的替代方案

| 方案 | 否决理由 |
|------|---------|
| 保留 QTE 百分比和倒计时 | 信息过多，像调试面板，不符合正式玩法提示 |
| 把跳过提示写进台词文本 | 污染剧情内容，未来改键或隐藏提示会难以维护 |
| 为每条叙事行配置跳过提示 | 把系统 UI 责任转嫁给内容制作，增加无意义字段 |
| 叙事开始时取消已有 QTE | 会掩盖玩家已经失衡的身体状态，削弱状态连续性 |
| 叙事期间摔倒但继续播放对话 | UI 与镜头反馈冲突，玩家难以理解当前优先状态 |
