# ADR 0014: Contextual Tutorial Prompts

**日期**: 2026-07-14
**状态**: Accepted

---

## 背景

教程不应在游戏开场一次性塞给玩家。BlindWalker 的核心体验来自玩家在黑暗中逐步学习移动、盲杖、踉跄、摔倒等反馈；因此教程应该在玩家实际遇到对应体验后出现。

现有系统已经通过 `EventBus` 广播移动、踉跄、摔倒、叙事和失败等事件。教程系统应监听这些事件并自行决定是否显示，而不是让 `GaitController`、`CutsceneManager`、`QtePromptUI` 等系统直接互相调用。

## 决策

### 事件驱动教程

引入上下文教程提示。教程由玩法事件触发，但触发端不直接控制教程 UI。

教程系统监听事件，判断教程是否已显示、是否已在栈中、当前是否满足安全显示条件，然后决定是否入栈或显示。

初始教程可以覆盖移动、盲杖、踉跄和摔倒，但 ADR 不把教程集合封闭化。后续可以继续增加触摸、触碰属性、交通规则等教程。

### 会话级一次性

每个教程在一次游戏运行会话中最多显示并确认一次。

玩家失败、回到 loading 页、重新开始同一运行会话时，已经确认过的教程不应再次出现。玩家关闭网页、刷新页面或重新访问游戏后，可以重新看到教程。

因此教程已读状态应保存在运行会话内，而不是写入磁盘、浏览器 local storage 或长期存档。

为支持场景 reload 后仍保留会话级已读状态，本 ADR 允许新增 `TutorialManager` Autoload。该单例只负责教程触发、栈、已读状态和安全显示判断，不承载玩法规则。

### 教程提示栈

教程提示采用栈型结构，而不是同屏显示多个教程。

当同一事件链触发多个教程时，后入栈的教程先显示。例如玩家先发生踉跄，随后摔倒，则起身后先显示摔倒教程；玩家关闭摔倒教程后，再显示踉跄教程。

同一个教程 ID 如果已经确认过、正在显示，或已经在栈中，则重复触发会被忽略。教程只有在玩家完成关闭确认后，才标记为已读。

### 显示时机

摔倒教程在第一次摔倒并起身后显示，而不是摔倒发生瞬间显示。摔倒过程本身应由镜头和身体反馈表达，教程不应削弱失控体验。

踉跄教程在第一次踉跄结束后显示，并用一条教程同时解释轻踉跄和失衡踉跄。若第一次失衡踉跄最终导致摔倒，则按教程栈规则处理：摔倒教程先显示，踉跄教程随后显示。

开场控制教程在进入 `PLAYING` 后触发；如果此时叙事正在播放，则教程入栈等待，叙事结束后再显示。开场教程只介绍基础移动和盲杖，不负责解释后续所有系统。

### UI 行为

教程 UI 不锁定玩家移动、视角、盲杖或触摸，不暂停世界，也不锁定 gameplay 后果。教程是说明性 UI，不是强制教学关卡。

教程 UI 放在屏幕左上角，以半透明卡片显示标题和短正文。卡片应避免遮挡屏幕中心、QTE 提示区域和底部叙事字幕。

教程通过长按 `TAB` 关闭。关闭符号使用常见的圆形加叉样式；按住 `TAB` 时，外层圆环显示关闭进度。松开 `TAB` 后进度清零。长按完成后关闭当前教程，并显示教程栈中的下一条。

### UI 优先级

教程 UI 与 QTE UI 同级并存，互不隐藏。QTE 位于中心偏下，教程位于左上角，两者职责不同且位置不冲突。这样可以避免玩家短按或松开 QTE 时导致教程反复出现和消失。

叙事 UI 高于教程 UI。叙事开始时，如果教程正在显示，则教程暂停隐藏；当前教程不标记为已读，并保留在栈顶。叙事结束后，教程系统重新判断是否可以显示。

失败/死亡 UI 高于教程 UI。失败或死亡时，教程隐藏；当前教程不标记为已读，并可在同一运行会话的后续重新开始中继续显示。

普通 HUD 和游戏世界低于教程 UI。

优先级概念为：

```text
失败/死亡 UI > 叙事 UI > 教程 UI == QTE UI > HUD / 游戏世界
```

### 安全显示条件

教程系统负责判断是否可以显示教程。安全显示条件至少包括：

- 当前没有叙事播放。
- 当前不是失败或成功结算。
- 当前没有正在显示另一个教程。
- 教程栈中存在待显示教程。

QTE 不阻止教程显示。

## 非目标

本 ADR 不实现教程内容全集，也不把当前教程 ID 集合固定为长期规范。

本 ADR 不实现长期持久化、不写 local storage、不提供“重置教程”设置。

本 ADR 不实现暂停世界、慢动作教学或强制教学关卡。

本 ADR 不改变 QTE、摔倒、踉跄、叙事或失败系统的核心状态机。

## 验收要求

实现阶段不能只启动游戏确认无报错。需要在 `scripts/debug/` 添加临时测试脚本，控制或模拟对应事件并跟踪 Godot 输出，至少确认：

- 进入 `PLAYING` 后基础教程入栈，并在叙事结束后显示。
- 第一次踉跄结束后显示综合踉跄教程。
- 第一次摔倒并起身后显示摔倒教程。
- 同一事件链中踉跄后摔倒时，摔倒教程先显示，关闭后再显示踉跄教程。
- 长按 `TAB` 才能关闭教程，短按不会关闭。
- 关闭后同一运行会话内同 ID 不再显示。
- 场景 reload 或失败后重新开始时，已确认教程仍不重复显示。
- 叙事开始会暂停教程显示，叙事结束后恢复。
- QTE 不会隐藏教程。
- 临时测试脚本完成后必须移除。

## 考虑的替代方案

| 方案 | 否决理由 |
|------|---------|
| 开场一次性显示所有教程 | 信息负担过重，玩家尚未遇到对应情境，学习效果差 |
| 在各玩法系统中直接弹教程 UI | 增加模块耦合，后续维护和扩展困难 |
| 摔倒瞬间暂停或慢动作教学 | 工作量高，影响物理、音频、相机和伤害计时，并削弱失控体验 |
| 教程锁定玩家输入 | 违背当前希望教程不打断游玩的方向 |
| 教程与 QTE 互斥显示 | 会在 QTE 短按/松开时造成教程忽隐忽现 |
| 长期持久化教程已读状态 | 当前短流程不需要，Web 存储和重置逻辑会增加额外复杂度 |
